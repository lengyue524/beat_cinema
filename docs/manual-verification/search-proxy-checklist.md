# 搜索代理手动验证清单

目标：验证歌曲搜索在 YouTube / Bilibili 下是否正确使用代理配置，并确认无回归。

## 前置条件

- 已设置有效的 BeatSaber 路径。
- 搜索面板可正常使用并可切换平台（YouTube / Bilibili）。
- 准备一个稳定关键词（建议英文歌名）。
- 准备一个可用自定义代理（例如 `127.0.0.1:7890`）。

## 用例 1：代理模式 = 不使用代理

步骤：
- 设置页将代理模式切换为“不使用代理”。
- 在 YouTube 搜索关键词。
- 在 Bilibili 搜索同一关键词。

预期：
- 搜索请求可发起并返回结果（或出现可理解错误提示）。
- 日志包含 `proxyMode=none` 且 `proxyApplied=false`。

## 用例 2：代理模式 = 自定义代理（有效地址）

步骤：
- 设置页切换为“自定义代理”，输入有效代理地址并保存。
- 在 YouTube 搜索关键词。
- 在 Bilibili 搜索同一关键词。

预期：
- 搜索可正常进行。
- 日志包含 `proxyMode=custom` 且 `proxyApplied=true`。

## 用例 3：代理模式 = 自定义代理（无效地址）

步骤：
- 保持“自定义代理”，填入无效地址并保存。
- 在 YouTube / Bilibili 发起搜索。

预期：
- 搜索失败时有用户可理解提示，不会卡住页面。
- 可继续修改配置并再次搜索。

## 用例 4：代理模式 = 系统代理

步骤：
- 切换为“系统代理”。
- 在 YouTube / Bilibili 发起搜索。

预期：
- 日志包含 `proxyMode=system`。
- 若系统代理可解析，则 `proxyApplied=true`；否则 `proxyApplied=false`。

## 用例 5：热切换立即生效

步骤：
- 在同一会话中按顺序切换：`none -> custom -> system`。
- 每次切换后立即搜索一次（YouTube / Bilibili 各至少一次）。

预期：
- 无需重启应用。
- 后续请求立即按新代理模式生效（日志字段随模式变化）。

## 回归检查

- 在搜索结果中执行“应用内播放”，确认功能正常。
- 在搜索结果中执行下载，确认下载链路功能正常。
- 播放/下载失败提示保持可理解，不出现明显交互阻塞。
